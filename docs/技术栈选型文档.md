## 2. 前端技术栈

### 2.1 核心选型

| 选型 | 理由 | 备选方案 |
|---|---|---|
| **React 19** | 项目已采用；React Compiler 自动 memo 提升复杂编辑器性能；生态最广 | Vue 3、Solid |
| **TypeScript 6** | 与 YAML Schema 强类型映射，降低跨端字段错配 | 纯 JS（不可取） |
| **Vite 8** | 冷启动毫秒级；Rolldown 集成已就绪；本项目已采用 | Next.js（SSR 暂不必要） |

### 2.2 UI 与样式

| 选型 | 用途 |
|---|---|
| **Tailwind CSS 4** | 原子化 CSS，与 UI 设计文档的 Token 体系直接映射 |
| **shadcn/ui** | 复制即用的高质量组件，避免 antd 等重型 UI 库的样式覆盖 |
| **Radix UI Primitives** | 无样式可访问性原语（被 shadcn 依赖），保证 a11y |
| **lucide-react** | 一致的图标体系，体积小 |
| **Framer Motion** | 场景卡片切换、AI 流式光标等微交互 |

### 2.3 状态与数据

| 选型 | 用途 | 说明 |
|---|---|---|
| **Zustand** | 全局 UI 状态（当前剧本、选中场景、编辑器视图模式） | 比 Redux 体积小 90%，对 YAML 大对象友好 |
| **TanStack Query** | 服务端状态（剧本列表、任务进度轮询） | 内置缓存、重试、SSE 适配 |
| **Immer** | YAML 对象不可变更新 | 编辑器内场景/节拍频繁增删改 |
| **valibot / zod** | 前端表单校验，与后端 Pydantic 模型对齐 | 优先 valibot（体积更小） |

### 2.4 编辑器与可视化

## 4. AI 与大模型层

### 4.1 模型选型

| 角色 | 模型 | 选型理由 |
|---|---|---|
| **主模型** | Claude Sonnet 4.6（`claude-sonnet-4-6`） | 长文推理强、中文质量高、工具调用稳定；支持 prompt caching 降低重复 system prompt 成本 |
| **高质量需求** | Claude Opus 4.7（`claude-opus-4-7`） | 高级订阅用户、复杂结构剧本（多人物长篇） |
| **成本优化** | DeepSeek-R1 | 国内调用快、价格低；用于免费版与 IP 评估批量任务 |
| **兜底降级** | Qwen 2.5 / GLM-4 国内开源 | 海外 API 不可达时本地化兜底 |

### 4.2 调用层抽象

```python
# 抽象 Provider 接口，便于动态路由
class LLMProvider(Protocol):
    async def complete(self, prompt: str, *, schema: dict | None) -> str: ...

# 路由策略：根据用户套餐 + 任务类型 + 实时成本选择
class Router:
    def pick(self, plan: Plan, task: TaskType) -> LLMProvider: ...
```

### 4.3 核心能力

- **Prompt Caching**：将 Schema 定义与 system prompt 复用，B 端高频调用节省 70%+ 输入成本
- **结构化输出**：通过 Claude tool use / DeepSeek JSON mode 强制返回符合 Schema 的结构
- **流式响应**：长章节通过 SSE 推送进度，前端实时显示生成状态
- **自动重试与修复**：Schema 校验失败时，将错误信息反馈给 LLM 进行二次修正
- **Token 预算控制**：单任务 Token 上限保护，避免成本失控

### 4.4 提示工程基础设施

| 工具 | 用途 |
|---|---|
| **Jinja2** | 提示词模板（多版本 A/B） |
| **Langfuse / 自建表** | 提示词版本管理与效果回溯 |
| **Promptfoo**（开发期） | 提示词回归测试 |

## 5. 数据存储层

### 5.1 PostgreSQL（Supabase 托管）

**选型理由**：
- 已在项目中引入（`requirements.txt` 中 `supabase`）
- 托管节省运维成本，自带 Auth / Storage / Realtime
- JSONB 字段适合存储 YAML 解析后的剧本结构
- 行级安全（RLS）便于多租户隔离

**核心表设计概要**：

| 表 | 关键字段 | 说明 |
|---|---|---|
| `users` | id, email, plan, quota_used | Supabase Auth 扩展 |
| `projects` | id, user_id, title, source_novel | 项目（一本小说=一个项目） |
| `scripts` | id, project_id, version, yaml_content::jsonb | 剧本 YAML 解析后存 JSONB |
| `tasks` | id, project_id, status, progress, error | 转换任务状态 |
| `chapters` | id, project_id, index, raw_text, token_count | 原始章节 |
| `usage_logs` | id, user_id, model, input_tokens, cost | 计费与配额 |
| `api_keys` | id, user_id, key_hash, scopes | B 端 API 鉴权 |
| `webhooks` | id, user_id, url, secret, events | 回调订阅 |

### 5.2 对象存储

| 内容 | 存储 | 说明 |
|---|---|---|
| 用户上传原始文件 (TXT/EPUB) | Supabase Storage | 国内访问可加挂阿里云 OSS 镜像 |
| 导出的 YAML / PDF 文件 | Supabase Storage | 带过期签名 URL |

### 5.3 Redis

| 用途 | Key Pattern |
|---|---|
| 任务队列 | `celery:queue:*` |
| 限流计数 | `ratelimit:{user_id}:{minute}` |
| LLM 结果缓存（章节级 hash） | `llm:cache:{sha256(prompt)}` |
| 会话/临时令牌 | `session:{token}` |

## 6. 任务队列与异步处理

### 6.1 选型

| 选型 | 理由 |
|---|---|
| **Celery 5 + Redis broker** | 生态成熟、运维资料多、支持优先级队列与重试策略 |
| **备选：ARQ** | 原生 async，代码更简洁；如果团队不依赖 Celery 周边工具可考虑切换 |

### 6.2 任务编排

```
用户提交 → API 创建 ScriptTask
        ↓
   主任务 split_chapters
        ↓ fan-out
   子任务 transform_chapter × N（并行）
        ↓ fan-in
   合并任务 merge_and_validate
        ↓
   写库 + 触发 Webhook
```

- **并行度**：默认每用户最多 5 章节并发，企业版可拉高
- **失败策略**：单章节失败重试 3 次，仍失败标记部分成功（其他章节正常返回）
- **超时**：单章节 5 分钟硬超时，整体 30 分钟

## 8. 可观测性与运维

| 维度 | 选型 | 说明 |
|---|---|---|
| 错误追踪 | **Sentry**（已在依赖中） | 前后端统一 |

- API P95 延迟 < 500ms（异步接口）
- 单章节转换平均耗时 < 3 分钟

- 系统可用性 ≥ 99.5%

## 9. 安全与合规

| 维度 | 措施 |
|---|---|
| **鉴权** | Supabase Auth + JWT；B 端 API Key（HMAC 签名）|

| **版权检测** | 文本指纹（SimHash）+ 与正版库比对 |
| **审计日志** | 全部 LLM 调用、付费操作、删除操作留痕 90 天 |
| **GDPR / 个保法** | 用户数据导出与删除接口（M9 前完成） |

## 11. 选型决策矩阵

### 11.1 关键三方案对比

| 决策点 | 候选 | 选择 | 关键理由 |
|---|---|---|---|
| 后端框架 | FastAPI / Django / Flask | **FastAPI** | async 原生、OpenAPI 自带、Pydantic 集成 |
| 数据库 | Self-host PG / Supabase / RDS | **Supabase** | 开发速度优先，未来可平滑迁出 |
| 任务队列 | Celery / ARQ / Dramatiq | **Celery**（备选 ARQ） | 生态成熟，监控工具齐全 |
| LLM 主供应商 | Claude / GPT / 国产 | **Claude（多模型路由）** | 中文长文质量稳定，prompt cache 控本 |
| 前端框架 | React / Vue / Svelte | **React 19** | 既有项目 + Monaco/编辑器生态最佳 |
| 样式方案 | Tailwind / CSS-in-JS / Sass | **Tailwind 4** | 与 UI 设计 Token 直接映射 |
| 部署平台 | 自建 K8s / Fly.io / Vercel | **分层：Vercel + Fly.io → ACK** | MVP 速度 → 规模化合规 |

### 11.2 主动放弃的选项

| 放弃 | 理由 |
|---|---|
| Next.js SSR | MVP 阶段无 SEO 需求，编辑器为重交互场景 |
| Microservices | 早期单服务即可，过早拆分增加复杂度 |
| 自研向量库 | 暂无 RAG 需求，避免运维负担 |
| 自建 LLM | 成本与质量短期内无法对标商业 API |

## 12. 演进路径

### M1~M3（MVP）

- 单体 FastAPI 服务 + Celery worker（同主机）
- Supabase 全家桶（DB + Auth + Storage）
- 单一 LLM Provider（Claude）
- Vercel + Fly.io 部署

### M4~M9（B 端增长）

- API 服务与 Worker 拆分独立部署
- 引入多模型路由 + Prompt Caching
- 接入内容安全 API
- 国内合规：阿里云镜像 + ICP 备案

### M10~M18（规模化）

- 迁移至阿里云 ACK（K8s）
- Worker 按章节量自动扩缩
- 数据分库（按租户）
- 海外节点（Cloudflare + Vercel 国际）

### M18+（生态）

- 开放 YAML Schema 标准，建立 Schema 版本治理委员会
- 与 AI 视频平台共建插件市场
- 多语种模型并行（zh-CN / en / ja）

## 附录 A · 风险与替代策略

| 选型 | 潜在风险 | 替代方案 |
|---|---|---|
| Supabase | 海外服务，国内合规与延迟 | 迁移到自建 PostgreSQL + MinIO + Casdoor |
| Claude API | 单点依赖、政策不确定 | 已设计多 Provider 路由，可热切到 DeepSeek/Qwen |
| Celery | 配置复杂、资源占用高 | 切换 ARQ（async 原生，代码量降低 40%） |
| Vercel | 海外节点国内偶发慢 | 阿里云 CDN 回源 + Static Export |
