## 1. 产品概述

### 1.1 一句话定位

**ScriptForge** 是面向中国网络文学作者及内容制作公司的 AI 剧本创作平台，能将 3 个章节以上的小说文本自动转换为结构化 YAML 格式剧本初稿，大幅降低 IP 改编门槛，提升微短剧剧本量产效率。

### 1.2 产品愿景

> 让每一部有潜力的网文，都能低成本地走向银幕。

当前小说改编剧本存在三大痛点：**剧作能力门槛高、改编周期长、格式标准不统一**。ScriptForge 通过 AI 能力将从"小说文本"到"可执行剧本初稿"的链路压缩到分钟级，让作者和制作方将精力聚焦在创意打磨而非格式转化上。

### 1.3 核心价值主张

| 用户角色     | 当前痛点                     | ScriptForge 提供的价值           |
| ------------ | ---------------------------- | -------------------------------- |
| 网文作者     | 不懂剧本格式，改编成本高     | 一键生成可编辑 YAML 剧本初稿     |
| 短剧制作公司 | 剧本产能不足，批量需求难满足 | API 批量接入，日产剧本量级提升   |
| 编剧工作室   | 前期拆本耗时，创意空间被压缩 | 自动完成结构拆解，编剧只需打磨   |
| IP 孵化机构  | 评估改编可行性成本高         | 快速生成样本集，辅助 IP 价值判断 |

## 2. 市场背景与机会

### 2.1 市场规模（数据驱动）

**中国网络文学 IP 生态正处于历史性爆发期：**

- 📊 **IP 改编市场**：2024 年中国网络文学 IP 改编市场规模达 **2985.6 亿元**，同比增长 14.61%（中国社会科学院，2025.05）
- 📊 **微短剧市场**：2024 年微短剧市场规模突破 **500 亿元**（部分数据来源显示达 800 亿），用户规模达 **6.96 亿**
- 📊 **内容供给**：2024 年网文 IP 改编短剧新增授权超 **1600 部**，同比增长 **81%**
- 📊 **创作者规模**：网文作者突破 **3119.8 万人**，作品总量达 **4165.1 万部**
- 📊 **海外增量**：中国网文出海市场规模超 **50 亿元**，海外用户超 **3.5 亿**，覆盖 200+ 国家

### 2.2 核心驱动因素

**微短剧的爆发为 AI 剧本工具打开结构性机会窗口：**

1. **低成本高频率**：短剧单部制作成本 30~300 万（对比长剧单集 100~1000 万），对剧本量产的需求极高
2. **改编加速**：平台方（抖音、快手、微信）主动扶持网文 IP 改编短剧，分账模式日趋成熟
3. **供给缺口**：编剧人才严重不足，内容供给跟不上市场扩张速度
4. **AI 工具渗透率提升**：阅文集团旗下"作家助手妙笔"集成 DeepSeek-R1 后，日活用户增长超 30%，印证 AI 辅助创作的用户接受度

### 2.3 市场切入点

```
主战场：微短剧剧本量产工具（B2B + B2C 双轮驱动）
次战场：长剧 IP 样本评估工具（B2B，服务影视公司）
未来战场：海外网文 IP 多语种剧本改编（出海增量）
```

## 3. 竞品分析

### 3.1 国际竞品

| 产品             | 核心功能             | 优势                     | 劣势                         | 对 ScriptForge 的启示      |

| **Melies**       | 小说/文章→screenplay | show-don't-tell 技巧强   | 无中文支持，无批量能力       | 中文本地化是核心护城河     |
| **Squibler**     | AI 全流程创作平台    | 功能全面，用户基数大     | 剧本模块偏西方格式，无 YAML  | 垂直深度 > 广度            |
| **NolanAI**      | AI 剧本写作软件      | 专注剧本，专业性强       | 无小说导入能力，无结构化导出 | 导入 + 结构化是双重壁垒    |

### 3.2 国内竞品

| 产品                | 核心功能         | 市场定位             | 局限                     |
| ------------------- | ---------------- | -------------------- | ------------------------ |
| **阅文妙笔**        | 网文 AI 续写辅助 | 面向在平台创作的作者 | 无剧本改编，无结构化导出 |

> **当前市场存在明确空白**：中文网文→结构化剧本（YAML 格式）的专业化工具几乎为零。ScriptForge 有机会成为该细分品类的定义者。

## 4. 目标用户画像

### Persona A：「量产型短剧公司」（B 端核心客户）

- **背景**：中小型短剧制作公司，月产短剧 3~10 部，编剧团队 2~5 人
- **痛点**：剧本产能是瓶颈，网文 IP 采购后转化耗时 1~2 周/部
- **需求**：批量接入 API，剧本初稿 10 分钟出稿，编剧只负责打磨
- **付费意愿**：高（节省人力成本显著，ROI 清晰）
- **使用场景**：上传采购网文 TXT → 批量生成多章节 YAML 剧本 → 分配编剧修订

### Persona B：「独立网文作者」（C 端种子用户）

- **背景**：在起点/番茄/七猫等平台创作，作品有一定读者基础，希望自主 IP 开发
- **痛点**：不懂剧本格式，找编剧改编费用高（数万~数十万），周期长
- **需求**：快速得到剧本初稿，了解改编可行性，低成本试水
- **付费意愿**：中等（有增收期望，但风险意识强）
- **使用场景**：粘贴自己章节文本 → 生成 YAML 初稿 → 自行微调后投稿短剧平台

### Persona C：「IP 孵化/版权机构」（B 端战略客户）

- **背景**：持有大量网文版权的机构（如阅文集团合作方、黑岩网等）
- **痛点**：需要低成本筛选高改编价值的 IP，人工评估效率极低
- **需求**：批量生成剧本样本，评估故事结构和视觉化潜力
- **付费意愿**：高（企业级采购）

## 7. 剧本 YAML Schema 设计规范

### 7.1 Schema 设计原则

| 原则             | 说明                                                       | 设计理由                                                         |
| ---------------- | ---------------------------------------------------------- | ---------------------------------------------------------------- |
| **机器可读优先** | 所有字段使用小写英文键名，值类型严格定义                   | 支持下游 AI 视频生成工具（Sora、可灵等）直接解析，构建工作流壁垒 |
| **人类可读兼顾** | 嵌套层级不超过 4 层，字段命名语义清晰                      | 编剧打磨阶段直接编辑 YAML，降低认知负担                          |
| **扩展性设计**   | `metadata.extensions` 字段预留自定义扩展空间               | 不同类型剧本（短剧/长剧/动漫剧本）可继承基础 Schema 扩展         |
| **国际化友好**   | 文本内容字段支持 `zh-CN`/`en` 双语嵌套                     | 服务出海需求，同一剧本可携带多语言版本                           |
| **语义完整性**   | 每个场景必须包含 `location`、`time`、`beats`，不允许空场景 | 确保 AI 生成内容满足基本剧本完整性标准，减少无效输出             |

### 7.2 完整 YAML Schema 定义

```yaml
# ScriptForge Script YAML Schema v1.0
# 适用于：电视剧、网络剧、微短剧剧本

script:
  # ────────────────────────────────────
  # 元数据块 (必填)
  # ────────────────────────────────────
  metadata:
    title: string                    # 剧本标题
    source_novel: string             # 原著小说名
    source_author: string            # 原著作者
    adapter: string                  # 改编者/公司名
    version: string                  # 剧本版本号，如 "1.0.0"
    created_at: datetime             # ISO 8601 格式，如 "2026-06-05T10:00:00+08:00"
    updated_at: datetime
    genre:                           # 类型标签（可多选）
      - enum: [ancient_romance, modern_romance, xianxia, urban_fantasy,
               suspense, comedy, historical, sci_fi, short_drama]
    format:
      type: enum                     # 剧本类型
        values: [tv_series, web_series, micro_drama_vertical,
                 micro_drama_horizontal, film, animation]
      total_episodes: integer        # 集数，微短剧通常 30~100 集
      episode_duration_min: integer  # 每集时长（分钟），微短剧通常 1~3 分钟
    language: string                 # 主要语言，如 "zh-CN"
    extensions: object               # 扩展字段，用于平台/业务自定义

  # ────────────────────────────────────
  # 人物档案块 (必填，至少 1 人)
  # ────────────────────────────────────
  characters:
    - id: string                     # 唯一标识，如 "char_001"，供场景引用
      name: string                   # 角色名
      aliases: [string]              # 别名/称谓列表
      role:
        type: enum                   # 角色类型
          values: [protagonist, deuteragonist, antagonist,
                   supporting, minor, narrator]
        arc: string                  # 人物弧线描述，如 "从自卑走向自我认同"
      traits: [string]               # 性格特征关键词，如 ["腹黑", "外冷内热"]
      relationships:
        - target_id: string          # 指向另一角色的 id
          type: enum
            values: [lover, enemy, family, friend, rival, mentor, subordinate]
          description: string        # 关系补充描述
      appearance: string             # 外貌描述（用于视觉参考）
      first_appears_in: string       # 首次出现的 scene_id

  # ────────────────────────────────────
  # 剧情大纲块 (可选，推荐填写)
  # ────────────────────────────────────
  outline:
    logline: string                  # 一句话故事概述（≤50字）
    acts:
      - act_number: integer          # 第几幕（1~4）
        title: string                # 幕名称
        summary: string              # 本幕概要
        conflict: string             # 核心矛盾
        episodes_range:
          start: integer
          end: integer

  # ────────────────────────────────────
  # 集数/章节块 (必填)
  # ────────────────────────────────────
  episodes:
    - episode_id: string             # 唯一标识，如 "ep_001"
      episode_number: integer        # 集号
      title: string                  # 本集标题
      source_chapters: [string]      # 对应原著章节，如 ["第1章", "第2章"]
      cold_open: string              # 片头钩子（可选，建议微短剧必填）
      scenes:

        # ────────────────────────────────
        # 场景块 (必填，每集至少 1 个场景)
        # ────────────────────────────────
        - scene_id: string           # 唯一标识，如 "sc_ep001_001"
          scene_number: integer      # 场景序号（集内递增）
          heading:
            interior_exterior: enum  # INT. / EXT. / INT./EXT.
              values: [INT, EXT, INT_EXT]
            location: string         # 地点名称，如 "皇宫御书房"
            time: enum               # 时间
              values: [DAY, NIGHT, DAWN, DUSK, CONTINUOUS, LATER, MOMENTS_LATER]
          characters_present: [string]  # 出现角色的 id 列表

          # 场景节拍列表（beats）
          beats:
            - beat_id: string        # 唯一标识，如 "bt_sc001_001"
              type: enum             # 节拍类型
                values: [action, dialogue, monologue, transition,
                         flashback, dream, title_card, sfx_note]

              # type = action 时
              action:
                description: string  # 动作描述，使用现在时主动语态
                visual_notes: string # 视觉化提示（镜头建议，可选）

              # type = dialogue 时
              dialogue:
                character_id: string # 说话角色 id
                parenthetical: string # 括号内动作说明，如 "(冷笑)" （可选）
                line: string         # 对白正文
                subtext: string      # 潜台词/创作意图注解（可选，面向编剧）

              # type = monologue 时（旁白/内心独白）
              monologue:
                character_id: string
                voice_over: boolean  # true = 画外音，false = 直接呈现
                line: string

              # type = transition 时
              transition:
                style: enum
                  values: [CUT_TO, FADE_TO, DISSOLVE_TO, SMASH_CUT, MATCH_CUT]
                target_scene_id: string  # 转场目标场景（可选）

          scene_notes: string        # 导演提示/编剧备注（可选）
          emotion_tone: enum         # 场景情绪基调
            values: [tense, romantic, comedic, melancholic,
                     suspenseful, action, dramatic, neutral]

      episode_ending_hook: string    # 集尾钩子/悬念（推荐填写，提升续看率）
      editor_notes: string           # 编辑备注（可选）
```

### 7.3 Schema 设计关键决策说明

| 决策                     | 选择                                            | 理由                                                                                |
| ------------------------ | ----------------------------------------------- | ----------------------------------------------------------------------------------- |
| **格式选型**             | YAML（非 JSON/XML）                             | YAML 对人类更可读，编剧可直接手动编辑；同时被主流 AI 工具广泛支持                   |
| **ID 引用体系**          | 所有实体用 ID 索引（character_id、scene_id 等） | 避免字符串重复，支持跨集引用，便于数据库存储和图谱分析                              |
| **beats 层级**           | 场景内细化到节拍（beat）级                      | 匹配微短剧高密度叙事需求；beat 级结构也是 AI 视频分镜的最小生成单元                 |
| **emotion_tone 字段**    | 显式标注情绪基调                                | 为 AI 配乐推荐、色调建议等后续工作流提供输入；也便于平台审核情绪分布                |
| **source_chapters 映射** | 每集映射回原著章节                              | 便于追溯改编忠实度，也方便书粉对照；B 端客户可用于版权存证                          |
| **extensions 扩展字段**  | 预留 metadata 和 scene 级扩展                   | 不同平台（抖音短剧、腾讯视频等）有各自的元数据字段需求，扩展字段避免频繁升级 Schema |
| **双语支持**             | 文本字段支持 zh-CN/en 嵌套                      | 出海场景（东南亚、日韩市场正高速增长）无需二次格式转换                              |

## 8. 技术架构概览

```
用户/API 层
    │
    ▼
┌─────────────────────────────────────────┐
│            API Gateway / BFF            │
│    (Rate Limiting · Auth · Logging)     │
└────────────────┬────────────────────────┘
                 │
    ┌────────────▼────────────┐
    │     任务队列 (Redis)     │
    │  Chapter → Task Shards  │
    └────────────┬────────────┘
                 │ 并行分发
    ┌────────────▼──────────────────────────┐
    │         AI 转换工作节点集群            │
    │                                       │
    │  Step 1: 文本预处理                   │
    │    - 章节切分 · 编码标准化 · 去噪      │
    │                                       │
    │  Step 2: 结构理解（LLM）              │
    │    - 人物识别 · 时间线提取             │
    │    - 场景边界检测                     │
    │                                       │
    │  Step 3: 剧本生成（LLM）             │
    │    - 内心独白→视觉化 · 对白优化       │
    │    - 场景头生成 · 节拍排布             │
    │                                       │
    │  Step 4: YAML 序列化 + Schema 校验    │
    │    - 字段校验 · ID 引用完整性检查     │
    │    - 自动修复常见错误                 │
    └────────────┬──────────────────────────┘
                 │
    ┌────────────▼────────────┐
    │      结果存储层          │
    │  PostgreSQL (结构化)     │
    │  OSS (原始文本/YAML文件) │
    └─────────────────────────┘
```

**关键技术选型：**

- **LLM 后端**：主模型使用 Claude claude-sonnet-4-20250514（长文推理强），辅以 DeepSeek-R1（成本优化），支持动态路由
- **Schema 校验**：使用 `jsonschema`（YAML 转 JSON 后校验），失败自动触发重试和修复
- **并发架构**：每个章节独立为一个 Task，支持 GPU 节点水平扩展，目标单集（30 分钟短剧）转换时间 < 5 分钟

## 9. 商业化模型

### 9.1 定价策略

采用 **「订阅制 + 用量包」混合模式**，以订阅制保证基础 MRR（月经常性收入），以用量包捕获 B 端爆发性需求。

#### C 端定价（网文作者）

| 套餐       | 价格    | 包含额度                     | 适用人群   |
| ---------- | ------- | ---------------------------- | ---------- |
| **免费版** | ¥0/月   | 3 次转换/月，仅前 3 章       | 体验用户   |
| **创作版** | ¥49/月  | 30 次转换/月，无章节限制     | 个人作者   |
| **专业版** | ¥149/月 | 120 次转换/月 + API 访问权限 | 高频创作者 |

#### B 端定价（制作公司 / 机构）

| 套餐           | 价格      | 包含额度                 | 附加服务                |
| -------------- | --------- | ------------------------ | ----------------------- |
| **团队版**     | ¥1,999/月 | 500 次转换/月，5 席位    | 专属客服 + SLA 保证     |
| **企业版**     | ¥9,999/月 | 3000 次转换/月，无限席位 | 专属模型微调 + 白标定制 |
| **API 用量包** | ¥0.1/章节 | 按量付费                 | 适合非稳定需求          |

> **定价逻辑**：节省一个编剧初稿工时（约 8~16 小时 × ¥200/小时 = ¥1600~3200），¥1999/月 的团队版 ROI 对 B 端客户极其清晰。

### 9.2 收入结构预测（18 个月）

| 阶段     | 时间    | 核心目标                            | 预计 MRR |
| -------- | ------- | ----------------------------------- | -------- |
| 冷启动期 | M1~M3   | 200 个 C 端付费用户 + 5 家 B 端试用 | ¥2 万    |
| 增长期   | M4~M9   | 1000 个 C 端 + 30 家 B 端           | ¥15 万   |
| 规模化期 | M10~M18 | 3000 个 C 端 + 100 家 B 端          | ¥60 万   |

### 9.3 增长飞轮

```
更多 B 端客户
      ↑
  收入增加 → 持续优化模型 → 输出质量更高
      ↑
  口碑传播 ← 作者转换成功 ← 更多 C 端用户注册
```

### 9.4 战略合作方向

1. **阅文集团 / 番茄小说**：平台内嵌入 ScriptForge 入口，作者一键改编，收入分成模式
2. **抖音/快手短剧平台**：成为官方认证短剧剧本工具，享受平台流量扶持
3. **AI 视频生成平台（可灵、即梦等）**：YAML Schema 直接对接，成为"小说→剧本→视频"全链路的标准中间件
4. **版权代理机构**：提供 IP 改编潜力评估服务，按报告收费
